# Tutorial: Runtime Events

Runtime events let the page observe what the player is doing without guessing from local state. This tutorial covers lifecycle events, subscribed frame updates, FPS reporting, layer taps, and poster-to-first-frame transitions.

## Listen to lifecycle events

The main lifecycle events are `ready`, `firstframe`, `start`, `repeat`, `update`, `fps`, `error`, and `completion`. Use them to drive logs, analytics, loading states, and playback UI.

`ready` means AnimaX has prepared the rendering model and platform rendering object. If you need frame updates, subscribe to the frames you care about after `ready`.

```tsx {3,9-17}
const handleReady = (e: AnimaXReadyEvent) => {
  pushLog('ready', e);
  invokeAnimaX('subscribeUpdateEvents', { frames: [0, 30, 60, 89] });
};

<animax-view
  id={ANIMAX_VIEW_ID}
  src={EVENTS_LOTTIE_SRC}
  fps-event-interval={500}
  bindready={handleReady}
  bindfirstframe={(e) => pushLog('firstframe', e)}
  bindstart={(e) => pushLog('start', e)}
  bindrepeat={(e) => pushLog('repeat', e)}
  bindupdate={(e) => pushLog('update', e)}
  bindfps={pushFPSLog}
  binderror={pushErrorLog}
  bindcompletion={(e) => pushLog('completion', e)}
/>;
```

## Subscribe to update events

`update` is different from the other lifecycle events: AnimaX only emits it for frames that have been subscribed. Subscribe in `ready` with `subscribeUpdateEvent` or `subscribeUpdateEvents`, then handle the callback with `bindupdate`.

## Handle layer taps

Use `bindtaplayers` when the animation itself should be interactive. The callback returns the tapped layer names, so the page can map animation layers to business actions.

Keep the mapping logic outside the animation component. The animation reports layer names, and the application decides what those names mean.

```tsx {1,9}
const handleTapLayers = (e: AnimaXTapEvent) => {
  setLayers(e.detail.layerList.join(', ') || 'no layer');
};

<animax-view
  src={EVENTS_TAP_LOTTIE_SRC}
  autoplay={true}
  loop={true}
  bindtaplayers={handleTapLayers}
/>;
```

## Switch from poster to first frame

For production pages, show a poster before the animation has rendered its first frame. Hide the poster in `firstframe` instead of `ready`; this avoids exposing a blank or partially prepared view.

`ready` is the right place to enable controls or prepare runtime data. `firstframe` is the right place to remove the poster.

```tsx {1,9-11}
const handleFirstFrame = (e: AnimaXFirstFrameEvent) => {
  setShowPoster(false);
  setStatus(`first frame: ${formatFrame(e.detail.current)}`);
};

<animax-view
  src={FIRSTFRAME_POSTER_LOTTIE_SRC}
  autoplay={true}
  bindready={handleReady}
  bindfirstframe={handleFirstFrame}
  binderror={handleError}
/>;

{
  showPoster ? (
    <view className="poster-layer">
      <image
        className="poster-image"
        src={DYNAMIC_REPLACEMENT_IMAGE_URL}
        mode="aspectFill"
      />
    </view>
  ) : null;
}
```

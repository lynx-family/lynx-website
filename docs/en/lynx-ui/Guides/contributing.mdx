# Contributing

Lynx UI welcomes code contributions from members of the Lynx community. We will carefully review all PRs and issues that meet our standards.

## Developing

### Setup

NodeJS: Install node version >= 20

Pnpm: Run `corepack enable`

```
npm install turbo --global

git clone git@code.byted.org:lynx/lynx-ui.git

cd lynx-ui

corepack enable

pnpm i
```

## Develop

Each component in Lynx UI has its own directory containing the following:

```bash
template
 ┣ __tests__    # Unit tests
 ┣ src          # Component code
 ┣ types        # Type definitions
 ┃ ┣ index.d.ts         # Component type definitions
 ┃ ┣ index.docs.d.ts    # Component type definition entry, including interface and type, for generating API Reference
 ┃ ┗ index.docs.zh.d.ts # Chinese component documentation type definition entry
 ┣ README.md    # Component documentation, mainly includes component description, Demo, etc.
 ┣ README.zh.md # Chinese version of component documentation
 ┣ edenx.config.ts
 ┣ package.json
 ┗ tsconfig.json
```

### Developing New Components

In the following documentation, we'll use creating a new `toast` component as an example

#### 1. Generate Component Directory

Execute the `makeNewComponent` command to generate the component directory

```
// In root directory
pnpm run makeNewComponent --create toast
```

You'll see a new `packages/lynx-ui-toast` directory created

#### 2. Complete Component Development

##### Step 1: Complete Component Type Definitions

Complete the component type definitions in the `packages/lynx-ui-toast/types` directory.

`index.docs.d.ts` contains all type definitions available for documentation generation. Usually includes `props` and `ref` definitions, as well as component entry functions.

`index.docs.zh.d.ts` contains all type definitions for Chinese documentation generation, needs to be synchronized with `index.docs.ts`.

`index.d.ts` will reexport all types from `index.docs.d.ts` and `index.docs.zh.d.ts`. Code implementation will depend on types defined in `index.d.ts`.

```typescript
// types/index.docs.d.ts
export interface ToastProps {}

export interface ToastRef {}

export type Toast = (props: ToastProps) => React.ReactElement;
```

```typescript
// types/index.d.ts
export * from './index.docs';
```

##### Step 2: Complete Component Code Development

Complete the component code development in the `packages/lynx-ui-toast/src` directory

```typescript
// src/index.ts
import type { Toast as ToastType, ToastProps, ToastRef } from "./index.docs";

const Toast: ToastType = () => {
  return <view>Toast</view>;
};

export { Toast };
export type { ToastProps, ToastRef };
```

#### 3. Add Component Examples

##### Step 1: Add Component Example Code

Add component example code in the `apps/examples/src` directory

```typescript
// In apps/examples/src/toast/Basic/index.tsx
import { Toast } from "@lynx-js/lynx-ui-toast";

const App = () => {
  return <Toast>Click me</Toast>;
};
```

##### Step 2: Add Example Entry

Add example entry in `apps/examples/lynx.config.mjs`

```javascript
// In apps/examples/lynx.config.mjs
{
  ...
  'toast-basic': './src/toast/Basic/index.tsx',
}
```

##### Step 3: Verify Component Example

Run examples to verify the code

```
// In root directory
turbo watch dev --filter 'examples'

// Or in packages/examples folder
turbo watch dev
```

#### 4. Complete Component Documentation

Component documentation is located in `packages/<component-name>/README.md`. The final documentation will be generated by combining `README.md` and `index.docs.d.ts`

Focus on the Guide section of the documentation, including component introduction, usage methods, examples, etc.

You can use `internal-website` components (like `Go`) in markdown

```markdown
<Go

  example="ActionSheet"
  defaultFile="Customized/index.tsx"
  defaultEntryFile="dist/ActionSheetCustomized/template.js"
  entry="Customized"
```

For details, refer to [internal-website](https://bits.bytedance.net/code/lynx/internal_website/)

#### 5. Add Tests

Basic element checks for components are done using headless lynx. Please add the component samples that need to be tested to TEST_COMPONENTS in apps/examples/tests/headless.test.ts.

Run tests locally using `pnpm run test`.

To update baseline files, please execute `pnpm run test --update`.

#### 6. Verify Package Dependencies

Run `pnpm dlx @manypkg/cli check` to check the lint of your package.json files. If it fails, run `pnpm dlx @manypkg/cli fix` to fix the lint.

Run `pnpm dlx sherif@latest` to check the dependencies. If it fails, run `pnpm dlx sherif@latest --fix` to fix the dependencies.

#### 7. Submit Code

### Generate Component Documentation

## MR Standards

- Each MR should have exactly 1 commit

## Submission Process

### Feature

> Usually used for new features, follows `master` branch release

1. Branch out from `master`, modify code
2. Run `pnpm changeset`, select packages to upgrade (choose patch or minor based on changes), write description
3. Submit code along with generated `changeset` file
4. Find Code Owner and Release Owner for review
5. Merge into `master` branch
6. Release Owner will release periodically (weekly)

### Hotfix

> Code Submitter

1. Branch out from `master`, modify code
2. Run `pnpm changeset`, select packages to upgrade, choose patch upgrade, write Change description
3. Submit code along with generated `changeset` file
4. Find Code Owner and Release Owner for review
5. Merge into `master` branch

> Release Owner

6. Periodically (daily or immediately) branch out from `master`
7. Run `pnpm changeset version`, check if version number changes meet expectations
8. Submit changes to `package.json` and `CHANGELOG.md`
9. Merge into `master` branch, luban automatically releases

## Version Release

### Snapshot Version

After submitting MR, CI will automatically publish Snapshot version

Snapshot versions can be used for testing, not recommended for production use.

Snapshot versions use different package names with `-canary` suffix, e.g., `@lynx-js/lynx-ui-button-canary` for `@lynx-js/lynx-ui-button`.

After merging, CI interface will show Snapshot version report, Lark will also receive notification
![ci_report](https://lf0-fast-deliver-inner.bytedance.net/obj/eden-internal/uhfyeh7nuhfnuhd/tmp/ci_report.jpeg)
![lark_report](https://lf0-fast-deliver-inner.bytedance.net/obj/eden-internal/uhfyeh7nuhfnuhd/tmp/lark_report.jpeg)

### Release Version

Release versions need to be released by Release Owner.

After features or bugfixes merge into master, CI will sync changes to `changeset-release/next` branch and generate version and changelog

Release Owner only needs to merge `changeset-release/next` branch, CI will automatically publish release version and Lark will receive notification

## FAQ

### How to lint code

lynx-ui uses biome and eslint for code linting. CI and local linting will be performed on changed code

> Changed code refers to code changes relative to the stable branch (master or next branch)

Code will be linted automatically when committing.

You can also use `pnpm check:changed` to check changed code

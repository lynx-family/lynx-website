When we open-sourced Lynx, we said that many production-proven capabilities—including a richer set of UI components—were [still on their way](/blog/lynx-unlock-native-for-more#this-is-not-the-end-but-a-new-beginning). From Lynx 3.6 to the current development version, we have kept moving that work forward: bringing more native elements into the open and using [lynx-ui](/blog/lynx-ui) to turn those foundations into components that frontend developers can use directly.

Now is a good time to look back at that progress and share how we are thinking about the next stage.

Lynx already has many capabilities proven in production, but it does not yet have a component ecosystem that is open, clear, and easy enough to participate in. What we want to build is not simply a longer component list. It is a complete path that can keep growing: cross-platform Elements as a public contract, **XElement extending the boundary of native capabilities**, and **lynx-ui organizing those capabilities into composable frontend components**. At the same time, real component scenarios in lynx-ui should feed back into Element semantics, XElement implementations, and cross-platform tests.

XElement and lynx-ui are not a one-way stack in which one disappears behind the other. They serve different developers and solve different problems, yet together determine whether Lynx can turn native experience into efficient, reusable cross-platform development capabilities.

## Why Elements Alone Are Not Enough

A native element can be powerful and still be difficult to use directly in a product. If platform differences, engine versions, performance switches, and compatibility strategies all leak into application code, every project has to learn and carry the same complexity. Without sound defaults and stable composition patterns, a richer lower-level stack can also create more repeated work.

The opposite extreme is no better. If every product interaction becomes another native property, the API surface keeps expanding while lower-level implementations struggle to keep pace with product needs. Frontend composition offers greater expressiveness, yet performance-sensitive gestures, animation, scrolling, and layering still need support from the main thread and native platform.

The goal of a component ecosystem is therefore not simply to wrap Elements. **XElement extends what Lynx can express natively; lynx-ui makes those capabilities stable, composable, and approachable for frontend developers.** When real components expose new performance, semantic, or compatibility requirements, they feed those needs back into Elements and XElement.

## From a Few Primitives to a Broader Set of Elements

In Lynx, [Elements](/guide/ui/elements-components) are the basic building blocks of an interface. `<view>`, `<text>`, and `<image>` provide the most fundamental structure and expression. Scrolling, input, media playback, and overlays require deeper cooperation with each platform's native UI system.

In Lynx 3.6, the open-source stack included a small set of extended native elements, primarily `<input>`, `<textarea>`, and `<overlay>`. Since then, we have gradually brought more capabilities already running in real applications into the open-source repository:

- Lynx 3.7 introduced the [`<svg>`](/api/elements/built-in/svg) element for cross-platform native vector rendering.
- Lynx 3.8 added interaction containers such as [`<refresh>`](/api/elements/built-in/refresh) and began opening up native Markdown rendering.
- In the current development version, capabilities such as `blur-view`, `markdown`, `scroll-coordinator`, `video`, `viewpager`, and `webview` are also entering the open stack across Android, iOS, and HarmonyOS.

Together, these elements cover several recurring needs: input and keyboards, independent overlay layers, vector and blur effects, refresh and complex scrolling, rich text and media, and embedded content. Many are not new experiments; they have been exercised in real products. Once their code is open, however, we still need to align APIs, close platform gaps, and bring documentation, examples, testing, and release processes into the open as well.

For that reason, “the code is open” is not the finish line. For capabilities that enter the official core set, we want to make their supported platforms, stability level, and known differences explicit, then uphold those commitments through public documentation, automated tests, and a clear versioning policy.

## XElement and lynx-ui: Better Together

Native elements solve the problems closest to the platform: reliable input, scrolling, gestures, layering, and rendering. A product-level component still needs state, composition, lifecycle, styling interfaces, and product-specific behavior.

[XElement](/guide/ui/elements-components#xelement) is Lynx's extended native element system. It lets capabilities that do not belong directly in the Engine—but still require deep platform integration—remain opt-in and evolve independently. [lynx-ui](https://github.com/lynx-family/lynx-ui) is the official headless ReactLynx component library maintained by the Lynx team. It does not decide what a component should look like. Instead, it organizes Elements, cross-platform behavior, and state into composable primitives that developers can use to build their own design systems.

The relationship is therefore neither one-to-one nor only bottom-up. XElement gives components new expressive power. lynx-ui brings that power into real product compositions and exposes new requirements in APIs, semantics, and platform consistency. Together, they form a feedback loop through shared Element contracts, usage scenarios, and tests.

## How Overlay Grows into a Family of Components

Overlay is a concrete example of this layering.

As part of XElement, the native `<overlay>` element solves an independent-layer rendering problem. It lets content escape the clipping and stacking constraints of the regular view tree and render in the appropriate native container. But `<overlay>` does not know what a dialog is, nor does it manage open and close state, enter and exit animation, anchor positioning, or drag-to-snap behavior.

lynx-ui first uses `OverlayView` to normalize how Overlay works across containers and platforms. From there:

- `Popover` composes anchor tracking, positioning, visibility state, and Presence;
- `Dialog` composes Trigger, Backdrop, Content, dismissal behavior, and enter and exit states;
- `Sheet` goes further by composing Overlay, Presence, main-thread gestures, and Motion to support dragging, resistance, and snap points.

This example also shows the ecosystem we want to build. A stable XElement can support multiple components. Components in lynx-ui then place more concrete demands on container selection, stacking, event pass-through, lifecycle, and animation coordination, helping `<overlay>` converge on clearer cross-platform semantics.

As XElement evolves, Popover, Dialog, and Sheet gain a more reliable native foundation. As those components reach more products, they expose gaps in the lower-level capability sooner. Developers should neither have to solve native layering and platform differences again in every project nor accept an opaque component that cannot be adapted.

The same loop appears in other forms. Long-list components need to absorb lower-level switches, platform differences, and version compatibility. Swiper needs programmable, low-latency main-thread interaction beyond a fixed native API. Sheet combines native layering and gestures with frontend state. Sometimes components absorb lower-level complexity; sometimes component scenarios expose a missing framework capability. Both directions move XElement and lynx-ui forward.

## What Belongs in the Official Core?

A thriving ecosystem does not mean that the official project implements every component. The larger the official set becomes, the more clearly we need to define its boundaries and maintenance responsibilities.

Whether the proposal is an XElement or a lynx-ui component, we believe three questions should guide whether it belongs in the officially maintained core:

1. Is it broadly needed across independent industries?
2. Can it serve as a foundation for several higher-level components?
3. Does it place high demands on performance, interaction latency, or native platform integration?

For a concrete proposal, we would also like to see real demand from at least three independent domains, clear compositional value, a maintainer prepared to take long-term responsibility, and comprehensive tests. For XElement, that also means explicit cross-platform semantics, native implementation boundaries, and compatibility requirements. For lynx-ui, it means composable APIs, openness to different design systems, and a sound abstraction over lower-level Element capabilities.

For the official set, a working implementation is not the finish line. Compatibility information, changelogs, migration guidance, runnable examples, and tests for performance, memory, and accessibility should gradually become part of the maintenance commitment.

These criteria still need to be tested in practice. We want to discuss them with the community: which native capabilities belong in the official XElement set, which interaction patterns lynx-ui should maintain over the long term, and what is better left to grow freely in the wider ecosystem?

The community can participate at different stages of this ecosystem:

- Share needs, problems, and feedback from real products;
- Contribute best practices, runnable examples, and cross-platform test cases;
- Join discussions about APIs, Element semantics, and test standards, and contribute fixes and new capabilities;
- Build community XElements, components, and tools for scenarios beyond the official core.

The Lynx team will continue to take responsibility for core XElement and lynx-ui implementations and their long-term maintenance. As development guidelines, testing, and distribution infrastructure improve, we also want to lower the barrier to building community XElements, industry components, and design systems, allowing both sides of the ecosystem to grow on a stable core.

## Bringing the Same Native Capabilities to More DSLs

lynx-ui currently targets ReactLynx. It is the official component library built on these lower-level capabilities, and a proving ground where real components validate Element semantics, interaction mechanisms, compatibility strategies, component APIs, and composition models. But the Lynx Engine and XElement are not tied to a single frontend framework. As more DSLs develop on Lynx, we do not want every ecosystem to begin by solving the same platform differences from scratch.

Ideally, different DSLs can share three kinds of foundations:

- **Reusable lower-level implementations:** mechanisms such as gesture arbitration, animation drivers, overlays, focus, and keyboard handling that depend on native code or the main thread;
- **Shared behavioral contracts and tests:** even when form semantics, state transitions, and accessibility behavior cannot share code directly, they should follow the same specification and pass the same test suite;
- **Thin, idiomatic wrappers:** each DSL can express composition, context, and controlled state in the way that feels most natural to its users.

This does not mean that today's lynx-ui components can run unchanged in every DSL. It means that XElement can remain a native capability ecosystem shared by multiple DSLs, while behavior specifications, component models, and tests proven in lynx-ui can help new DSLs avoid repeating the same work. At the same time, we can gradually move framework-independent behavior lower in the stack, making each DSL's component wrappers thinner and more consistent.

## Consistent Semantics, Without Pretending Platforms Are Identical

Cross-platform consistency does not require every platform to use an identical implementation.

When a capability can fully follow Web semantics, we will prefer APIs that are familiar to Web developers. When the Web does not accurately express native behavior, we will start from native interaction patterns and user expectations across platforms to define consistent core semantics. Where complete unification is not practical, we favor progressive enhancement while documenting compatibility and fallback behavior clearly.

That takes more than a TypeScript definition. Cross-platform Element semantics must be validated both by XElement implementations across platforms and by real component compositions in lynx-ui. Component APIs, compatibility matrices, and automated tests will become important infrastructure connecting both sides of the ecosystem.

## What Is Next

A component ecosystem is difficult to build from a dated checklist. We prefer to describe the work ahead in three horizons: **Now, Next, and Exploring**.

### Now: Open and Stabilize Core Capabilities

- Bring more production-proven XElements into the open-source stack and continue aligning behavior across Android, iOS, and HarmonyOS;
- Maintain lynx-ui core components in parallel so new native capabilities become components frontend developers can use directly;
- Improve documentation, runnable examples, and interactive previews on both sides, with clearer platform support and stability information for every capability;
- Gradually improve compatibility matrices, changelogs, and migration guides so upgrade paths become more transparent;
- Keep validating and refining Element APIs, interaction mechanisms, and defaults through real component scenarios.

### Next: Establish Standards We Can Share

- Define cross-platform Element semantics and compatibility baselines;
- Improve development, integration, testing, and release standards for XElement;
- Continue refining component APIs and composition models in lynx-ui;
- Establish proposal, admission, and maintenance processes for both core XElements and core components;
- Gradually make performance, memory, and accessibility part of the verifiable quality baseline for core capabilities;
- Open more shared cross-platform test infrastructure and contribution workflows.

### Exploring: Let the Ecosystem Cross Framework Boundaries

- Explore discovery, distribution, and dependency resolution for both XElements and frontend components;
- Continue moving shareable, framework-independent behavior lower in the stack;
- Let more DSLs reuse the same XElements while developing idiomatic component wrappers and shared tests;
- Lower the barrier for the community to build native extensions, industry components, and design systems.

Some of these directions are already in progress; others are questions we still need to answer with the community. As the designs become more concrete, we will continue to share progress through RFCs, documentation, and implementation updates.

## Building Shared Infrastructure for Cross-Platform UI

The vitality of the Web ecosystem has never come only from how many controls browsers ship. It comes from stable primitives, shared standards, open tools, and the space for anyone to keep building. We want the Lynx ecosystem to have that foundation while preserving the distinct value of native rendering, cross-platform delivery, and low-latency interaction.

That requires both directions to grow together. XElement lets new native capabilities enter Lynx and converge on shared semantics across platforms. lynx-ui makes those capabilities available to developers through familiar, composable component APIs. They validate and push each other forward; neither is merely an accessory to the other.

As that cycle takes shape, a native capability or component no longer needs to be a one-off implementation owned by one platform, framework, or product team. It can be expressed through different DSLs, composed into different design systems, and preserve consistent user semantics across platforms. Cross-platform native UI can become shared infrastructure that the whole ecosystem accumulates together, rather than an engineering cost paid repeatedly by every team.

We especially want to discuss three questions with the community:

1. What cross-platform semantics should an Element promise, and how should XElement validate them?
2. How should component APIs and composition models reflect and drive lower-level capabilities?
3. What should an XElement or component need to demonstrate before it enters the officially maintained core?

GitHub Discussions will be the primary home for proposals, standards discussions, and the decisions that follow.

Open-sourcing Lynx made native-rendered cross-platform capabilities available to more developers. In the next stage, we want to open up the complete evolution path from native extensions to frontend components. Our goal is not only to provide more XElements or more components, but to let them reinforce each other and build, together with the community, a cross-platform UI ecosystem for the open Web community with a truly native experience.

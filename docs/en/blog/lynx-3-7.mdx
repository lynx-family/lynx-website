Lynx 3.7 is now officially released!

This release brings official desktop platform support for **MacOS** and **Windows**, expands **Lynx for AI** with open-sourced skills and better observability, and updates **ReactLynx** with simpler External Bundle presets and cross-thread shared modules.

Let's take a closer look at what's new in Lynx 3.7.

## Desktop Support: MacOS and Windows

We [previously planned](/blog/lynx-open-source-roadmap-2025.html) to open source desktop support in Lynx 3.5, and [later decided](/blog/lynx-3-5.html#one-more-thing-desktop-support-preview) to go further and open source the full desktop stack. With Lynx 3.7, that vision is now a reality.

Lynx officially supports **MacOS** and **Windows** as first-class platforms. You can now build and render Lynx views inside native desktop applications, using the same front-end stack and tooling you already use for mobile and web.
The desktop UI is self-rendered by Lynx rather than composed from native platform widgets.
We also plan to open source Lynxtron, an Electron-like desktop application framework that uses Lynx as its UI renderer.

### Full Rendering Engine on Desktop

The same rendering engine that powers Lynx on mobile is now available on MacOS and Windows with near-complete feature parity.

**CSS: 97% property coverage.** 311 out of 319 supported Lynx CSS properties are now available on desktop, including:

- **Layout**: Flexbox, Grid, Linear Layout, the full box model, positioning, and relative positioning.
- **Styling**: typography, borders, shadows, gradients, masks, filters, and [CSS custom properties](/api/css/properties/css-variable).
- **Animation**: `animation`, `transition`, `@keyframes`, and the full transform stack.

**Elements: 73% attribute coverage.** 179 out of 246 element attributes are already available on desktop, with identical support on MacOS and Windows. Most built-in elements work directly on desktop today.

In practice, most Lynx pages built for mobile can already render on desktop with minimal or no front-end changes.

You can check the detailed compatibility status for every API on the [API Status Dashboard](/help/dashboard), which now includes **MacOS** and **Windows** columns.

### Getting Started

For platform-specific integration details, see [Integrate with Existing Apps](/guide/start/integrate-with-existing-apps), which now includes desktop guides alongside Android, iOS, HarmonyOS, and Web.
A playground app built with Lynxtron is also expected to provide a more direct way to fetch and run Lynx examples on PC.

### Desktop-Specific Capabilities

Beyond rendering parity, Lynx 3.7 adds capabilities designed specifically for the desktop environment, including mouse, keyboard, and wheel events, as well as CSS cursor support.

#### Mouse Events

Lynx 3.7 now supports most desktop [mouse event](/api/lynx-api/event/mouse-event) capabilities.

```tsx
<view
  onMouseDown={(e) =>
    console.log(`Button ${e.button} at (${e.clientX}, ${e.clientY})`)
  }
  onMouseMove={(e) => handleHover(e)}
  onMouseLeave={() => resetState()}
/>
```

#### Keyboard Events

Lynx 3.7 also adds desktop [keyboard event](/api/lynx-api/event/key-event) support.

```tsx
<input
  onKeyDown={(e) => console.log(`Key down: ${e.key}`)}
  onKeyUp={(e) => console.log(`Key up: ${e.key}`)}
/>
```

#### Wheel Events

Lynx 3.7 also adds desktop [wheel event](/api/lynx-api/event/wheel-event) support.

```tsx
<view onWheel={(e) => console.log(`Wheel delta: ${e.deltaX}, ${e.deltaY}`)} />
```

#### CSS `cursor`

Lynx 3.7 adds [CSS `cursor`](/api/css/properties/cursor) support on desktop so you can present clearer pointer feedback for interactive UI.

```tsx
<view style={{ cursor: 'pointer' }}>
  <text>Hover me</text>
</view>
```

## SVG: Scalable Vector Graphics Element

SVG is now available as a standalone Lynx element in 3.7, bringing native vector rendering across platforms for icons, charts, and other resolution-independent graphics.

`<svg>` parses vector nodes on a dedicated thread and supports [most SVG elements and attributes](/api/elements/built-in/svg), with consistent rendering behavior across Android, iOS, HarmonyOS, MacOS, Windows, and Web.

## Lynx for AI: Open-Sourced Skills

Lynx 3.7 continues the **Lynx for AI** work introduced in 3.6, with more day-to-day workflows now available as open-source agent skills.

The community skill collection at [/ai](/ai) can now be installed directly with:

```bash
npx -y skills add lynx-community/skills -s <skill-name>
```

The released skills cover three common areas:

- **Page inspection and debugging** with DevTool Skill and [Lynx DevTool MCP](/ai/lynx-devtool-mcp).
- **TypeScript and ReactLynx guidance** with setup, typing, event handling, and best-practice knowledge.
- **Tracing and template tooling** with trace recording, trace analysis, TASM codec, and bytecode debug info mapping.

The main improvement here is observability: coding agents can now inspect running pages, verify interactions, and debug their own changes with tighter feedback loops.

## ReactLynx Updates

Lynx 3.7 brings two practical improvements for ReactLynx projects: simpler External Bundle setup and easier code reuse across the background thread and main thread.

### External Bundle Presets

ReactLynx projects can now enable common external bundle settings with a single preset:

```js
import { pluginExternalBundle } from '@lynx-js/external-bundle-rsbuild-plugin';

export default defineConfig({
  plugins: [
    pluginExternalBundle({
      externalsPresets: {
        reactlynx: true,
      },
      externals: {
        './App.js': 'comp-lib.template.js',
      },
    }),
  ],
});
```

See the [External Bundle guide](/rspeedy/external-bundle) and the [`pluginExternalBundle`](/api/rspeedy/external-bundle-rsbuild-plugin.pluginexternalbundle) API reference for more details.

### Cross-Thread Shared Modules

Shared modules make it easier to reuse plain logic in both runtimes through [Import Attributes](/api/react/Document.import-attributes):

```js
import { myHelper } from './utils.js' with { runtime: 'shared' };

function App() {
  useEffect(() => {
    myHelper();
  }, []);

  function handleScroll() {
    'main thread';
    myHelper();
  }
}
```

This is a good fit for pure logic such as math utilities and formatting helpers. The module is bundled into both runtimes, but state remains isolated in each thread.

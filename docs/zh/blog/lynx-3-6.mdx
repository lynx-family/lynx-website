# Lynx 3.6: reactlynx-use, Lynx for AI, CSS for Design

<BlogAvatar list={['liujilong', 'Dugyu', 'lynx']} />

Lynx 3.6 已经正式发布！

本次版本引入 reactlynx-use 以提升开发体验，推出 LLM 工具集以增强 AI 原生开发能力，并围绕设计表达能力增强了 CSS 能力。

在原生侧，Lynx 3.6 进一步完善平台集成，在 HarmonyOS 上支持独立的 `BackgroundRuntime` 与异步创建 `TemplateBundle` ，并在 iOS 上新增 LynxView 的 Auto Layout 支持。

下面我们一起来看看 Lynx 3.6 带来了哪些新变化。

## `reactlynx-use`

[`reactlynx-use`](https://github.com/lynx-community/reactlynx-use) 是一套专为 ReactLynx 生态打造的 React 风格 Hooks 库。

`reactlynx-use` 提供与 React Hooks 一致的调用范式，与 ReactLynx 项目深度集成，并专为 Lynx 独特的跨平台能力而构建。内置近 30 个经开源社区验证的实用 Hooks（如 `useEffectOnce`、`useDebounce` 等），还提供了一组面向 Lynx 双线程架构的 Hooks 能力。

主要包括：

- **useVelocity**：追踪主线程 Tap 事件的速度与方向。
- **useTapLock**：将主线程 Tap 事件限制在指定的方向与范围内。
- **useMainThreadImperativeHandle**：`useImperativeHandle` 的 Lynx 主线程版本。
- **useEventListener**：提供更灵活的事件监听能力，支持 `exposure`、`disexposure` 等。

这些 Hooks 使得在理解并直接利用 Lynx 底层运行时模型的同时，依然可以保持符合 React 直觉的开发体验。

## Lynx for AI

大语言模型通常基于公开网络数据训练，对最新的 Lynx 能力和最佳实践缺乏直接认知。为此，Lynx 团队与社区正在共同建设一套工具体系，帮助 AI 更好地理解 Lynx 的 API、架构约束与推荐用法。

这一方向为 AI 辅助 Lynx 开发奠定了基础，使模型能够在 Lynx 的语境中进行更准确的推理与生成。

### Lynx Docs MCP

[llms.txt](https://llmstxt.org/) 是一种帮助大语言模型将网站内容作为可检索的知识库的标准。

Lynx 网站已完整支持该标准，对于任意页面，只需将其 `.html` 扩展名替换为 `.md`，即可获取原始 Markdown。例如：

- [https://lynxjs.org/llms.txt](https://lynxjs.org/llms.txt)
- [https://lynxjs.org/react/introduction.md](https://lynxjs.org/react/introduction.md)

此外，[Lynx Docs MCP](/ai/lynx-docs-mcp) 提供了 `llms.txt` 并配套精心设计的提示词，使你的编码代理（如 Gemini、Claude、Cursor 或 Copilot）能够高效访问 Lynx 文档，从而更好地支持开发任务。

### Lynx Devtool MCP

[Lynx DevTool MCP](/ai/lynx-devtool-mcp) 是一个 MCP 服务器，帮助编码代理连接设备并像 DevTools 一样检视与操控 Lynx 页面：读取元素/控制台/源码、执行点击拖拽等交互，并支持截图给多模态模型分析。

## CSS for Design

在 Lynx 3.6 中，CSS 在设计系统中承担了更加明确的角色。
CSS 作为一种表达设计意图的媒介，既描述设计决策如何在不同层级中被组合与生效，也直接参与界面中视觉结构与感知的形成。

本次版本围绕 **CSS 变量、渐变与滤镜** 三个方面，增强了 CSS 的表达能力。

### 用于可计算样式的 Inline CSS Variables

在 Lynx 3.6 中，CSS 变量可以以内联方式定义和引用。这使 CSS 变量能够直接应用在**组件边界处**(JS 中定义的参数在这里由 CSS 解析为具体样式)，而无需依赖祖先节点的 class 切换（如 `theme-light` / `theme-dark`），也不需要命令式的 JS `setProperty` 调用。

当组件的样式由用户选择或在运行时计算得到时，这种能力尤为关键，因为此类连续值无法通过有限的预定义 class 集合来表达，否则就只能退回到命令式的样式更新方式。

```tsx
style={{
  '--x': pointer.x,  // normalized parameter from JS
  backgroundColor: 'var(--dot-accent-color)', //defer resolution to component CSS
}}
```

在这种情况下，Inline CSS Variables（内联 CSS 变量）通过 JS 将设计参数交由 CSS 的继承与计算体系处理，使设计意图能够被动态解析与表达，并在不破坏组件作用域隔离的前提下，维持声明式的样式模型。

### 用于设计变量分层的 Nested CSS Variables

在 Lynx 3.6 中，CSS 变量可以基于其他变量进行定义，从而支持 [Design Token](https://m3.material.io/foundations/design-tokens/overview)（设计变量）的分层组合与逐步解析。

Design Token 用于承载高层级的设计决策，例如颜色、排版、间距、阴影与光影效果。通过 Nested CSS Variables（嵌套 CSS 变量），这些设计决策可以逐步解析为组件级变量，并最终落实为具体的视觉样式。

```css
/* Resolve and compose shadow tokens via nested CSS variables. */
.card {
  --blur: 12px;
  --ambient-shadow: 0 0 var(--blur) var(--color);
  box-shadow: var(--ambient-shadow), var(--rim-shadow);
}
```

这一能力在处理 `box-shadow` 这类复合声明时尤为关键。一个 `box-shadow` 往往包含多个阴影层，每一层又由位移、模糊、扩散和颜色等多个相互关联的维度共同定义。借助 Nested CSS Variables，阴影层现在可以由 Design Token 派生并进行组合，并在 CSS 中保持清晰、可追溯的结构关系。

### 作为设计表面的 Conic Gradients

Lynx 3.6 新增支持 `conic-gradient()` 这一 CSS 图像函数。它可以用于支持 `<image>` 类型值的 CSS 属性中，例如 `background-image` 和 `mask-image`，用于在 CSS 中直接生成连续的角度型颜色过渡。

```css
.bg {
  background-image: conic-gradient(from 30deg, #ff7385, #fe69a1, #00d0f1);
}
```

圆锥渐变非常适合用于表达角度或周期性结构，例如颜色环、色相选择器等。在实际场景里，它可以作为一种轻量的设计表面，被组合进交互控件与可视化工具中。

### 用于形态生成的 CSS Filters

Lynx 3.6 扩展了 CSS filter（滤镜）能力，新增支持 `brightness`、`contrast` 与 `saturate`。

CSS `filter` 属性不再只是用于细微的视觉调节，而是可以被组合进形态生成的过程之中。在 Lynx 3.6 中，新增支持的 `contrast` 滤镜与既有的 `blur` 能力配合使用，可以放大重叠区域的视觉响应，使原本柔和、分散的形状逐渐融合为连贯的整体。
例如形状融合、感知分组以及流动边界，而无需依赖位图资源或额外的自定义渲染管线。

通过这种组合方式，开发者可以直接通过 CSS 构建诸如形状融合、感知分组以及流动边界等视觉结构，而无需依赖位图资源或额外的自定义渲染管线。

## Native APIs

### HarmonyOS 支持独立的 BackgroundRuntime

HarmonyOS 新增支持独立的 [BackgroundRuntime](/api/lynx-native-api/lynx-background-runtime)，与 Android 和 iOS 的现有能力进一步对齐。

借助独立的 BackgroundRuntime，开发者可以在不依赖渲染的情况下独立运行后台 JS 任务，或提前创建后台运行时，以降低 LynxView 初始化阶段的延迟。

### HarmonyOS 平台新增异步创建 TemplateBundle 的方法

在 Android 和 iOS 平台上，[`TemplateBundle.fromTemplate`](/api/lynx-native-api/template-bundle/from-template) 可以在任意线程执行；而在 HarmonyOS 上，ArkTS 逻辑必须运行在应用主线程。

为减少主线程 I/O 阻塞，Lynx 3.6 在 HarmonyOS 上引入了异步创建 `TemplateBundle` 的 API：

- [`TemplateBundle.fromTemplateAsync`](/api/lynx-native-api/template-bundle/from-template-async)
- [`TemplateBundle.fromTemplateAsyncWithOption`](/api/lynx-native-api/template-bundle/from-template-async-with-option)

### iOS 平台 LynxView 新增 enableAutoLayout 能力

iOS 平台新增方法 [`[LynxView enableAutoLayout]`](/api/lynx-native-api/lynx-view/enable-auto-layout)。

开启后，LynxView 可以直接从客户端的 Auto Layout 约束中解析排版参数，无需再手动设置 LynxView 的布局配置。

Lynx 开源之初，我们曾提到，许多经过生产验证的能力——包括更丰富的 UI 组件——[还在路上](/blog/lynx-unlock-native-for-more#这不是终点而是新的起点)。从 Lynx 3.6 到当前开发版本，我们一直在推进这项工作：让更多原生元件进入开放环境，也通过 [lynx-ui](/blog/lynx-ui) 将这些底层能力组合成前端开发者可以直接使用的组件。

现在，是时候回顾这段进展，也分享我们对下一阶段的思考了。

Lynx 已经拥有许多经过生产验证的能力，但尚未形成一个足够开放、清晰、易参与的组件生态。我们希望补上的，不只是组件数量，而是**原生能力与前端组件之间缺失的中间层**：以跨平台 Element 为公共契约，以官方维护的核心组件为基础，让原生质量的 UI 能力可以被不同 DSL 组合、复用和共同演进。

## 从少数原语到更完整的 Element 集合

在 Lynx 中，[元件（Element）](/guide/ui/elements-components)是界面的基础构建单元。`<view>`、`<text>` 和 `<image>` 负责最基本的结构与表达；滚动、输入、媒体播放和浮层等能力，则需要与各平台的原生 UI 系统深入协作。

Lynx 3.6 时，开源体系中的扩展原生元件主要包括 `<input>`、`<textarea>` 与 `<overlay>`。此后，我们逐步将更多已经在真实应用中运行的能力带入开源仓库：

- Lynx 3.7 引入了跨平台原生矢量渲染元件 [`<svg>`](/api/elements/built-in/svg)。
- Lynx 3.8 继续补充 [`<refresh>`](/api/elements/built-in/refresh) 等交互容器，并开始开放 Markdown 渲染能力。
- 在当前开发版本中，`blur-view`、`markdown`、`scroll-coordinator`、`video`、`viewpager` 与 `webview` 等能力也在 Android、iOS 和 HarmonyOS 上陆续进入开放体系。

这些元件覆盖了几类常见需求：输入与键盘、独立浮层、矢量与模糊效果、刷新与复杂滚动、富文本与媒体，以及嵌入式内容。它们中的许多并不是新的实验，而是经历过真实业务检验的实现。开源之后，我们还需要继续统一 API、补齐平台差异，并将文档、示例、测试和发布流程一同带到开放环境中。

因此，“代码已经开放”并不是工作的终点。对于进入官方核心集合的能力，我们希望逐步明确它支持哪些平台、处于什么稳定性等级、有哪些已知差异，并通过公开文档、自动化测试与版本策略维护这些承诺。

## Element 还不是组件

原生元件解决的是最靠近平台的一层问题：如何获得可靠的输入、滚动、手势、层级与渲染能力。但一个产品级组件还需要处理状态、组合方式、生命周期、样式接口以及产品特定的交互。

这也是 [lynx-ui](https://github.com/lynx-family/lynx-ui) 存在的意义。lynx-ui 是由 Lynx 团队维护的 ReactLynx 无样式组件库。它不规定组件应该长什么样，而是把跨平台行为与状态组织成可组合的 primitive，让开发者能够在其上构建自己的设计系统。

目前，Lynx 的 UI 能力大致可以从三个层次理解：

| 层次            | 主要职责                                              | 例子                                         |
| --------------- | ----------------------------------------------------- | -------------------------------------------- |
| Engine 内置元件 | 提供最基础、与引擎生命周期紧密相关的渲染原语          | `<view>`、`<text>`、`<image>`、`<list>`      |
| 扩展原生元件    | 按需提供依赖平台能力或高性能交互的原生原语            | `<input>`、`<overlay>`、`<svg>`、`<refresh>` |
| lynx-ui         | 组合元件、状态与行为，形成面向 ReactLynx 的无样式组件 | `Popover`、`Dialog`、`Sheet`、`Swiper`       |

扩展原生元件目前主要通过 XElement 组织和集成。它的包结构与集成形式未来仍可能演进，但背后的分层会继续存在：不是所有能力都必须进入 Engine，也不是所有原生能力都应该由业务自行实现。我们需要一层可以按需集成、同时具备官方质量保障的原生能力集合。

## 从 Overlay 长出一组组件

Overlay 是这套分层的一个具体例子。

原生 `<overlay>` 解决的是独立渲染层问题：它可以让内容脱离普通视图树的裁剪与层级限制，并在正确的原生容器中显示。但 `<overlay>` 本身并不知道什么是对话框，也不负责管理打开、关闭、进出场动画、锚点定位或拖拽吸附。

lynx-ui 首先通过 `OverlayView` 统一不同容器和平台上的 Overlay 使用方式。在此之上：

- `Popover` 组合锚点追踪、定位、开关状态与 Presence；
- `Dialog` 组合 Trigger、Backdrop、Content、关闭行为与进出场状态；
- `Sheet` 进一步组合 Overlay、Presence、主线程手势与 Motion，支持拖拽、阻尼和吸附点。

## 官方应该维护哪些核心能力

一个繁荣的生态并不意味着官方实现所有组件。相反，官方集合越大，边界和维护责任就越需要清晰。

我们认为，一个能力是否应该进入官方维护的核心集合，可以优先考虑三个问题：

1. 它是否被多个独立行业普遍需要？
2. 它是否能成为若干上层组件的基础？
3. 它是否对性能、交互延迟或原生平台协作有较高要求？

对于具体提案，我们还希望看到来自至少三个独立领域的真实需求、清晰的上层组合价值，以及愿意长期负责的维护者和完整测试。这个标准本身还需要在实践中验证，我们也希望与社区一起讨论：什么应该成为所有 Lynx 开发者可以依赖的公共基础，什么更适合留在开放生态中自由生长。

官方团队将继续承担核心实现和长期维护。社区不仅可以反馈缺少什么，也可以参与需求发现、API 评审、跨平台语义与测试标准的定义。对于设计系统、行业组件和更具体的产品形态，我们希望稳定的核心能力能够为社区留下足够大的创造空间。

## 让组件能力不止属于一种 DSL

lynx-ui 目前面向 ReactLynx，但 Lynx Engine 并不绑定某一种前端框架。随着更多 DSL 在 Lynx 上发展，我们不希望每个生态都从平台差异开始重新解决同样的问题。

理想情况下，不同 DSL 可以共享三类基础：

- **可以复用的底层实现**：例如手势竞争、动画驱动、浮层、焦点与键盘等依赖原生或主线程的机制；
- **共同的行为契约与测试**：即使表单语义、状态转换和无障碍行为无法直接共用代码，也应该遵循相同规范并通过同一组测试；
- **符合各自习惯的轻量封装**：组件组合、上下文和受控状态等 API，由不同 DSL 使用最自然的方式表达。

这并不意味着现有 lynx-ui 组件可以原封不动地运行在所有 DSL 上。它意味着我们可以逐步把框架无关的行为沉淀到更低层，把 Element、行为规范和测试套件变成共同资产，让每一种 DSL 的组件封装更薄、更一致。

## 一致的语义，而不是假装平台没有差异

跨平台一致不意味着每个平台的实现必须完全相同。

当一个能力可以完整遵循 Web 语义时，我们会优先使用 Web 开发者熟悉的 API；当 Web 无法准确表达原生行为时，我们会从各平台的原生交互与用户预期出发，定义一致的核心语义。对于无法完全统一的部分，我们倾向于通过渐进增强保留平台能力，并清晰标注兼容性状态和降级行为。

这需要比一份 TypeScript 类型定义更多的工作。Element 的跨平台语义、组件 API 与组合模型、兼容性矩阵和自动化测试，都会成为接下来组件体系的重要基础设施。

## 下一步

组件生态的建设很难由一张带有固定日期的清单完成。我们更愿意把接下来的工作分为 **Now、Next 与 Exploring** 三个层次。

### Now：继续开放并稳定核心能力

- 将更多经过生产验证的 Element 带入开源体系；
- 持续改善 Android、iOS 与 HarmonyOS 的行为一致性；
- 维护 lynx-ui 的核心组件，补充文档、示例和测试；
- 为每项能力提供更清晰的平台支持与稳定性信息。

### Next：建立可以共同遵循的标准

- 定义 Element 的跨平台语义与兼容性基线；
- 完善组件 API 与组合模型；
- 建立核心组件的提案和准入机制；
- 开放更多跨平台测试基础设施与贡献流程。

### Exploring：让生态跨越框架边界

- 继续下沉可以共享的框架无关行为；
- 为更多 DSL 提供轻量组件封装和共同测试；
- 探索组件的发现、分发与兼容性管理方式。

这些方向中，有些已经在推进，有些仍然是需要与社区共同回答的问题。我们会在形成更具体的设计后，继续通过 RFC、文档和实现更新分享进展。

## 一起建立跨端 UI 的公共基础

Web 生态的活力从来不只来自浏览器内置了多少控件，更来自稳定的原语、共同的标准、开放的工具，以及任何人都可以在其上继续创造的空间。我们希望 Lynx 的组件生态也能拥有这样的基础，同时保留原生渲染、跨平台交付和低延迟交互带来的独特价值。

当这些基础逐渐建立，一个组件就不再是某个平台、某个框架或某个团队内部的一次性实现。它可以被不同 DSL 表达，被不同设计系统组合，并在多个平台上保持一致的用户语义。跨端原生组件也可以从反复发生的工程成本，变成整个生态能够共同积累的公共资产。

我们尤其希望与社区一起讨论三个问题：

1. 组件 API 与组合模型应该如何设计？
2. 一个组件应该满足什么条件，才能进入官方维护的核心集合？
3. Element 应该承诺怎样的跨平台语义？

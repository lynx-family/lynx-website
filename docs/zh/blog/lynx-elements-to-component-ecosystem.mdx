Lynx 开源之初，我们曾提到，许多经过生产验证的能力——包括更丰富的 UI 组件——[还在路上](/blog/lynx-unlock-native-for-more#这不是终点而是新的起点)。从 Lynx 3.6 到当前开发版本，我们一直在推进这项工作：让更多原生元件进入开放环境，也通过 [lynx-ui](/blog/lynx-ui) 将这些底层能力组合成前端开发者可以直接使用的组件。

现在，是时候回顾这段进展，也分享我们对下一阶段的思考了。

Lynx 已经拥有许多经过生产验证的能力，但尚未形成一个足够开放、清晰、易参与的组件生态。我们希望补上的，不只是组件数量，而是一个能够持续生长的完整链路：以跨平台 Element 为公共契约，由 **XElement 扩展原生能力的边界**，再由 **lynx-ui 将这些能力组织成可组合的前端组件**。与此同时，lynx-ui 中的真实组件场景也会反向推动 Element 语义、XElement 实现和跨平台测试继续演进。

XElement 与 lynx-ui 并不是谁隐藏在谁背后的单向分层。它们面向不同的开发者、解决不同的问题，又共同决定 Lynx 能否把原生体验转化为高效、可复用的跨端开发能力。

## 为什么仅有 Element 还不够

一个原生元件可以足够强大，却仍然难以直接用于产品。平台差异、引擎版本、性能开关和兼容策略如果全部暴露给业务，每个项目都需要重新学习并承担同一份复杂度。底层能力越丰富，如果缺少合理默认值与稳定的组合方式，这种重复成本也可能越高。

但另一个极端同样不可取：如果每一种产品交互都被固化为新的原生属性，API 表面积会不断膨胀，底层实现也很难跟上产品需求。前端组合提供了更高的表达力；而性能敏感的手势、动画、滚动与层级协作，又需要主线程和原生能力的支持。

因此，组件生态的目标并不是简单地给 Element 包一层。**XElement 扩展 Lynx 能够原生表达什么；lynx-ui 让这些能力能够以稳定、可组合、前端友好的方式被持续使用。**当真实组件暴露出新的性能、语义或兼容性需求时，它们又会反向推动 Element 与 XElement 演进。

## 从少数原语到更完整的 Element 集合

在 Lynx 中，[元件（Element）](/guide/ui/elements-components)是界面的基础构建单元。`<view>`、`<text>` 和 `<image>` 负责最基本的结构与表达；滚动、输入、媒体播放和浮层等能力，则需要与各平台的原生 UI 系统深入协作。

Lynx 3.6 时，开源体系中的扩展原生元件主要包括 `<input>`、`<textarea>` 与 `<overlay>`。此后，我们逐步将更多已经在真实应用中运行的能力带入开源仓库：

- Lynx 3.7 引入了跨平台原生矢量渲染元件 [`<svg>`](/api/elements/built-in/svg)。
- Lynx 3.8 继续补充 [`<refresh>`](/api/elements/built-in/refresh) 等交互容器，并开始开放 Markdown 渲染能力。
- 在当前开发版本中，`blur-view`、`markdown`、`scroll-coordinator`、`video`、`viewpager` 与 `webview` 等能力也在 Android、iOS 和 HarmonyOS 上陆续进入开放体系。

这些元件覆盖了几类常见需求：输入与键盘、独立浮层、矢量与模糊效果、刷新与复杂滚动、富文本与媒体，以及嵌入式内容。它们中的许多并不是新的实验，而是经历过真实业务检验的实现。开源之后，我们还需要继续统一 API、补齐平台差异，并将文档、示例、测试和发布流程一同带到开放环境中。

因此，“代码已经开放”并不是工作的终点。对于进入官方核心集合的能力，我们希望逐步明确它支持哪些平台、处于什么稳定性等级、有哪些已知差异，并通过公开文档、自动化测试与版本策略维护这些承诺。

## XElement 与 lynx-ui：相辅相成

原生元件解决的是最靠近平台的一层问题：如何获得可靠的输入、滚动、手势、层级与渲染能力。但一个产品级组件还需要处理状态、组合方式、生命周期、样式接口以及产品特定的交互。

[XElement](/guide/ui/elements-components#xelement) 是 Lynx 的扩展原生元件体系。它让不适合直接进入 Engine、又需要深入平台实现的能力可以按需集成和独立演进。[lynx-ui](https://github.com/lynx-family/lynx-ui) 则是由 Lynx 团队维护的 ReactLynx 无样式组件库。它不规定组件应该长什么样，而是把 Element、跨平台行为与状态组织成可组合的 primitive，让开发者能够在其上构建自己的设计系统。

因此，它们之间不是简单的一一对应，也不是只有从底层到上层的单向依赖。XElement 为组件提供新的表达能力；lynx-ui 把这些能力带入真实的产品组合，并暴露出 API、语义和平台一致性上的新需求。二者通过共同的 Element 契约、使用场景和测试形成反馈循环。

## 从 Overlay 长出一组组件

Overlay 是这套分层的一个具体例子。

作为 XElement 的一部分，原生 `<overlay>` 解决的是独立渲染层问题：它可以让内容脱离普通视图树的裁剪与层级限制，并在正确的原生容器中显示。但 `<overlay>` 本身并不知道什么是对话框，也不负责管理打开、关闭、进出场动画、锚点定位或拖拽吸附。

lynx-ui 首先通过 `OverlayView` 统一不同容器和平台上的 Overlay 使用方式。在此之上：

- `Popover` 组合锚点追踪、定位、开关状态与 Presence；
- `Dialog` 组合 Trigger、Backdrop、Content、关闭行为与进出场状态；
- `Sheet` 进一步组合 Overlay、Presence、主线程手势与 Motion，支持拖拽、阻尼和吸附点。

这个例子也说明了我们所期待的组件生态：一个稳定的 XElement 可以支撑多个上层组件；lynx-ui 中的组件又会对容器选择、层级、事件穿透、生命周期和动画协同提出更具体的要求，反过来帮助 `<overlay>` 形成更清晰的跨平台语义。

随着 XElement 演进，Popover、Dialog 和 Sheet 可以获得更可靠的原生基础；随着这些组件在更多产品中被使用，我们也能更早发现底层能力的缺口。开发者既不必在每个项目中重新处理原生层级和平台差异，也不必接受一个无法定制的组件黑盒。

同样的循环还会以不同形式出现。长列表组件需要收敛底层开关、平台差异与版本兼容；Swiper 需要在固定原生 API 之外获得可编程、低延迟的主线程交互；Sheet 则需要把原生浮层、手势体系与前端状态组合在一起。有时，组件吸收底层复杂度；有时，组件场景会暴露新的框架能力缺口。正是这两个方向共同推动着 XElement 与 lynx-ui 演进。

## 什么应该进入官方核心

一个繁荣的生态并不意味着官方实现所有组件。相反，官方集合越大，边界和维护责任就越需要清晰。

无论是 XElement 还是 lynx-ui 组件，一个能力是否应该进入官方维护的核心集合，都可以优先考虑三个问题：

1. 它是否被多个独立行业普遍需要？
2. 它是否能成为若干上层组件的基础？
3. 它是否对性能、交互延迟或原生平台协作有较高要求？

对于具体提案，我们还希望看到来自至少三个独立领域的真实需求、清晰的组合价值，以及愿意长期负责的维护者和完整测试。对于 XElement，这还意味着明确的跨平台语义、原生实现边界和兼容性要求；对于 lynx-ui，这意味着可组合的 API、对不同设计系统的开放性，以及对底层 Element 能力的合理抽象。

对于官方集合，功能实现完成也不是交付终点。兼容性信息、变更记录、迁移指导、可运行示例，以及性能、内存和无障碍相关的测试，都应该逐步成为维护承诺的一部分。

这些标准本身还需要在实践中验证，我们也希望与社区一起讨论：哪些原生能力应该进入官方 XElement 集合，哪些交互模式应该由 lynx-ui 长期维护，什么又更适合留在开放生态中自由生长。

社区可以从不同阶段参与这套生态的建设：

- 分享来自真实产品的需求、问题与反馈；
- 贡献最佳实践、可运行示例与跨平台测试案例；
- 参与 API、Element 语义和测试标准的讨论，并贡献修复与新能力；
- 构建社区 XElement、组件和工具，探索官方核心之外更广阔的场景。

官方团队将继续承担核心 XElement 与核心 lynx-ui 组件的实现和长期维护。随着开发规范、测试和分发基础逐步完善，我们也希望降低构建社区 XElement、行业组件和设计系统的门槛，让两侧生态都能在稳定核心之上继续生长。

## 让同一套原生能力服务更多 DSL

lynx-ui 目前面向 ReactLynx，是承载这些底层能力的官方组件库，也是我们通过真实组件验证 Element 语义、交互机制、兼容策略、组件 API 与组合模型的验证场。但 Lynx Engine 和 XElement 并不绑定某一种前端框架。随着更多 DSL 在 Lynx 上发展，我们不希望每个生态都从平台差异开始重新解决同样的问题。

理想情况下，不同 DSL 可以共享三类基础：

- **可以复用的底层实现**：例如手势竞争、动画驱动、浮层、焦点与键盘等依赖原生或主线程的机制；
- **共同的行为契约与测试**：即使表单语义、状态转换和无障碍行为无法直接共用代码，也应该遵循相同规范并通过同一组测试；
- **符合各自习惯的轻量封装**：组件组合、上下文和受控状态等 API，由不同 DSL 使用最自然的方式表达。

这并不意味着现有 lynx-ui 组件可以原封不动地运行在所有 DSL 上。它意味着 XElement 可以继续作为多个 DSL 共享的原生能力生态；lynx-ui 中已经验证过的行为规范、组件模型和测试，也可以帮助新的 DSL 少走弯路。与此同时，我们还可以逐步把框架无关的行为沉淀到更低层，让每一种 DSL 的组件封装更薄、更一致。

## 一致的语义，而不是假装平台没有差异

跨平台一致不意味着每个平台的实现必须完全相同。

当一个能力可以完整遵循 Web 语义时，我们会优先使用 Web 开发者熟悉的 API；当 Web 无法准确表达原生行为时，我们会从各平台的原生交互与用户预期出发，定义一致的核心语义。对于无法完全统一的部分，我们倾向于通过渐进增强保留平台能力，并清晰标注兼容性状态和降级行为。

这需要比一份 TypeScript 类型定义更多的工作。Element 的跨平台语义需要在 XElement 的多平台实现中得到验证，也需要在 lynx-ui 的真实组件组合中接受检验。组件 API、兼容性矩阵和自动化测试，会成为连接两侧生态的重要基础设施。

## 下一步

组件生态的建设很难由一张带有固定日期的清单完成。我们更愿意把接下来的工作分为 **Now、Next 与 Exploring** 三个层次。

### Now：继续开放并稳定核心能力

- 将更多经过生产验证的 XElement 带入开源体系，持续改善 Android、iOS 与 HarmonyOS 的行为一致性；
- 同步维护 lynx-ui 的核心组件，让新增原生能力能够形成前端开发者可直接使用的组件；
- 补充两侧的文档、可运行示例和交互式预览，为每项能力提供更清晰的平台支持与稳定性信息；
- 逐步完善兼容性矩阵、变更记录和迁移指南，让升级路径更加透明；
- 通过真实组件场景持续验证和收敛 Element API、交互机制与默认策略。

### Next：建立可以共同遵循的标准

- 定义 Element 的跨平台语义与兼容性基线；
- 完善 XElement 的开发、集成、测试与发布规范；
- 继续打磨 lynx-ui 的组件 API 与组合模型；
- 建立覆盖核心 XElement 与核心组件的提案、准入和维护机制；
- 将性能、内存与无障碍逐步纳入核心能力的可验证质量基线；
- 开放更多两侧共用的跨平台测试基础设施与贡献流程。

### Exploring：让生态跨越框架边界

- 探索 XElement 与前端组件的发现、分发和依赖解析机制；
- 继续下沉可以共享的框架无关行为；
- 让更多 DSL 复用同一套 XElement，并发展符合各自习惯的组件封装和共同测试；
- 降低社区构建原生扩展、行业组件和设计系统的门槛。

这些方向中，有些已经在推进，有些仍然是需要与社区共同回答的问题。我们会在形成更具体的设计后，继续通过 RFC、文档和实现更新分享进展。

## 一起建立跨端 UI 的公共基础

Web 生态的活力从来不只来自浏览器内置了多少控件，更来自稳定的原语、共同的标准、开放的工具，以及任何人都可以在其上继续创造的空间。我们希望 Lynx 的生态也能拥有这样的基础，同时保留原生渲染、跨平台交付和低延迟交互带来的独特价值。

这需要两个方向同时成长：XElement 让新的原生能力可以进入 Lynx，并在多个平台上形成共同语义；lynx-ui 则让这些能力通过熟悉、可组合的组件接口真正到达开发者手中。它们相互验证、相互推动，任何一侧都不是另一侧的附属品。

当这个循环逐渐建立，一个原生能力或组件就不再是某个平台、某个框架或某个团队内部的一次性实现。它可以被不同 DSL 表达，被不同设计系统组合，并在多个平台上保持一致的用户语义。跨端原生 UI 也可以从反复发生的工程成本，变成整个生态能够共同积累的公共资产。

我们尤其希望与社区一起讨论三个问题：

1. Element 应该承诺怎样的跨平台语义，XElement 又应该如何验证这些语义？
2. 组件 API 与组合模型应该如何反映和推动底层能力？
3. 一个 XElement 或组件应该满足什么条件，才能进入官方维护的核心集合？

Lynx 开源让更多开发者能够使用原生渲染的跨端能力；下一阶段，我们希望进一步打开这些能力从原生扩展到前端组件的完整演进过程。我们的目标不只是提供更多 XElement 或更多组件，而是让两者相辅相成，与社区一起建立一个属于开放 Web 开发者、同时拥有原生体验的跨端 UI 生态。

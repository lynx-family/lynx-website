# 什么是 Lynxtron？

Lynxtron 是一个使用 [Lynx](https://lynxjs.org/) 和 Node.js 构建桌面应用的框架。如果你了解 [Electron](https://www.electronjs.org/)，可以将 Lynxtron 理解为**把 Chromium 渲染层替换为 Lynx 的 Electron**。

- **主进程**侧本质上就是 [Electron](https://www.electronjs.org/docs/latest/tutorial/process-model#the-main-process)。Node.js 控制应用生命周期、窗口、菜单、托盘、对话框等系统级 API。
- **UI** 侧是 [Lynx](https://lynxjs.org/zh/guide/start/quick-start.html) 运行时。可以使用 ReactLynx、VueLynx 等 Lynx 运行时 DSL 编写界面，最终使用 [`<view>`](/api/elements/built-in/view)、[`<text>`](/api/elements/built-in/text)、[`<image>`](/api/elements/built-in/image) 等原生元素进行渲染，而非 HTML。

:::tip 已经熟悉 Electron 和 Lynx？
那你已经掌握了 Lynxtron 的大部分知识。本页只介绍**新的部分**：那些只因 Electron + Lynx 组合才存在的概念。
:::

## 架构：单进程，双线程隔离

与 Electron 的多进程模型（主进程 + 渲染进程）不同，Lynxtron 运行在**单个进程**中，通过**主线程 + Lynx 后台线程**形成和 Electron 双进程模型对偶的双线程隔离；其中 Lynx 后台线程可以被多个窗口共享，也可以为窗口使用独立后台线程。

<img src={lynxtronArchitecturePng} alt="Lynxtron 单进程双线程架构" />

| Lynxtron 线程     | 线程关系           | Electron 对偶                                 | 主要职责                                                                                                                                         |
| ----------------- | ------------------ | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **主线程**        | 每个应用一个       | Electron 主进程 + Chromium 渲染侧的主线程职责 | Node.js 主进程 API、窗口管理、原生能力、Lynx [MTS](/react/main-thread-script)、输入、布局、渲染                                                  |
| **Lynx 后台线程** | 可共享或按窗口独立 | Electron Renderer 中的页面业务 JS             | 直接运行在 Node.js 上，承载各窗口隔离的 BTS Context，执行 Lynx UI 业务逻辑；BTS Context 不是 Node.js 全局环境，可通过桥接访问白名单 Node.js 能力 |

Lynxtron 在单进程内，把主控能力和 Lynx 主线程能力放在主线程，把 Lynx UI 业务逻辑放在 Lynx 后台线程。Lynx 后台线程直接运行在 Node.js 上，但每个窗口的 BTS Context 是隔离 context，不是 Node.js 全局环境，不能直接使用 Node.js 接口。多个窗口可以共享同一个后台线程，也可以为窗口分配独立后台线程；无论哪种方式，各窗口的 BTS Context 仍然相互隔离；BTS 只能通过 `NativeModules.bridge` 或 [preload scripts](/lynxtron/learn/Communication-Between-Node-And-Lynx#使用-preload-scripts) 暴露的白名单能力访问 Node.js。

## Lynxtron 中的关键差异

以下机制用于连接 Electron-like 主进程模型和 Lynx UI 运行时，是 Lynxtron 相比 Electron 或纯 Lynx 需要额外理解的部分：

### 使用 Lynx 元素构建桌面 UI

Lynxtron 为 Lynx 的 UI 系统增加了桌面特有的能力：

- **[`LynxWindow`](/lynxtron/api/@lynx-js/lynxtron/Class.LynxWindow.mdx) 替代 Electron [`BrowserWindow`](https://www.electronjs.org/zh/docs/latest/api/browser-window)**：窗口加载 Lynx bundle，并使用 Lynx 元素渲染 UI，而非加载 HTML 页面
- **[`-x-app-region: drag`](/lynxtron/learn/Custom-Window-Styles#定义-ui-拖拽区域)**：让 Lynx 元素成为窗口拖拽区域（类似 Electron 的 [`-webkit-app-region: drag`](https://www.electronjs.org/zh/docs/latest/tutorial/custom-window-interactions)）
- **[`<title-bar-view>`](/api/elements/built-in/title-bar-view)**：用于自定义标题栏的内置 Lynx 元素

→ [了解更多：构建自定义窗口](/lynxtron/learn/Custom-Window-Styles)

### 线程间通信

Lynxtron 有三种线程间通信模式，对应 [Electron 的 IPC](https://www.electronjs.org/zh/docs/latest/tutorial/ipc)：

| 方向                       | Lynxtron                                                                                                              | Electron 对应                                                                                                                                                       |
| -------------------------- | --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Node.js → Lynx**         | [`LynxWindow.sendGlobalEvent()`](/lynxtron/api/@lynx-js/lynxtron/Class.LynxWindow.mdx#sendglobalevent)                | [`webContents.send()`](https://www.electronjs.org/zh/docs/latest/api/web-contents)                                                                                  |
| **Lynx → Node.js**（双向） | `NativeModules.bridge.call` → [`-lynx-invoke`](/lynxtron/api/@lynx-js/lynxtron/Class.LynxWindow.mdx#L3199-3200) 事件  | [`ipcRenderer.invoke()`](https://www.electronjs.org/zh/docs/latest/api/ipc-renderer) + [`ipcMain.handle()`](https://www.electronjs.org/zh/docs/latest/api/ipc-main) |
| **Lynx → Node.js**（单向） | `NativeModules.bridge.send` → [`-lynx-message`](/lynxtron/api/@lynx-js/lynxtron/Class.LynxWindow.mdx#L3220-3221) 事件 | [`ipcRenderer.send()`](https://www.electronjs.org/zh/docs/latest/api/ipc-renderer) + [`ipcMain.on()`](https://www.electronjs.org/zh/docs/latest/api/ipc-main)       |

→ [了解更多：Node 与 Lynx 之间的通信](/lynxtron/learn/Communication-Between-Node-And-Lynx)

### 通过 [Preload Scripts](/lynxtron/learn/Communication-Between-Node-And-Lynx#使用-preload-scripts) 集成 Node.js

[Preload scripts](/lynxtron/learn/Communication-Between-Node-And-Lynx#使用-preload-scripts) 可以理解为 [Electron preload scripts](https://www.electronjs.org/zh/docs/latest/tutorial/tutorial-preload) / contextBridge 模式面向 Lynx BTS 的能力桥接层：preload scripts 在具备 Node.js 能力的隔离 context 中执行，再把明确允许的方法或对象暴露给 Lynx UI。

与 Electron 不同，Lynxtron 的 preload scripts 和 BTS 位于同一进程内的隔离 JS context 中。[`contextBridge.exposeInLynxBTS()`](/lynxtron/api/@lynx-js/lynxtron/context-bridge/Interface.ContextBridge.mdx#exposeinlynxbts) 暴露的不是单纯的可序列化消息 payload，而是可供 BTS 直接调用的 JS 对象；这些对象可以包含函数、闭包，也可以返回 Promise。

核心流程是：

1. 在 [`LynxWindow`](/lynxtron/api/@lynx-js/lynxtron/Class.LynxWindow.mdx) 上配置 [`lynxPreference.preload`](/lynxtron/api/@lynx-js/lynxtron/Interface.LynxPreference.mdx#preload) 指定 preload script 文件。
2. preload script 直接使用 Node.js API，并通过 [`contextBridge.exposeInLynxBTS()`](/lynxtron/api/@lynx-js/lynxtron/context-bridge/Interface.ContextBridge.mdx#exposeinlynxbts) 暴露白名单能力。
3. Lynx BTS 通过 [`NativeModules.nodejs.exposed`](/lynxtron/api/@lynx-js/lynxtron/lynx/Interface.LynxtronNodejs.mdx#exposed) 调用暴露出来的对象。

BTS 所在的 Lynx 后台线程直接运行在 Node.js 上，但 BTS 仍然处于隔离 context，不是 Node.js 全局环境，不能直接访问 Node.js 接口；所有 Node.js 能力都需要经过 preload scripts 显式暴露。

→ [了解更多：Lynxtron 中 Node.js 与 Lynx 的通信](/lynxtron/learn/Communication-Between-Node-And-Lynx)
